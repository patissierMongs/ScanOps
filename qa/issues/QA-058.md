# QA-058 — GUI _stop sets _user_stopped on an already-exited process (120ms drain-window race), mislabeling a genuine failure / rc=2 validation error as a user 'stop' with a false resume promise

- **group**: R6   **disposition**: FIX   **severity**: low   **dimension**: ui-ux / result-reliability
- **dedup_key**: `user-stopped-race-on-exited-proc-mislabels-failure`
- **location**: scanner/scanops_scanner_gui.py _stop (guards only `if self.proc is None`) + final_status_text branch order (user_stopped checked before rc==2)

## Evidence
`_stop` guards only on `if self.proc is None: return` then unconditionally sets `self._user_stopped = True` — it never checks `proc.poll()` (unlike `_force_kill`). `self.proc` is cleared only in `_drain_output`'s done-handler, which polls on a 120ms timer. The worker enqueues `('done', rc)` after the process has already exited, so for up to ~120ms `self.proc` stays non-None and Stop stays enabled. A Stop click in that window (a natural reaction to a scan that just failed) passes the guard and sets `_user_stopped=True` on an exited process, so the pending `('done', rc)` drains through `final_status_text(rc, ..., user_stopped=True)`:
- rc=1 (genuine failure) → "중지됨 ... [재개 실행]" instead of "실패(종료 코드 1)" — a real failure hidden as a clean user stop.
- rc=2 (pre-exec validation error, no state) → also the stop/resume message, re-introducing the false-resume-for-rc=2 case QA-032 removed (the `user_stopped` branch precedes the `rc==2` branch).

This is the inverse of QA-047/054 (which correctly label a genuine stop); here a NON-stop outcome is mislabeled, so it is not covered by them. Introduced by the round-5 QA-054 `_user_stopped` flag.

## Proposed fix
1. `_stop`: `if self.proc is None or self.proc.poll() is not None: return` (mirror the `_force_kill` idiom) so a finished-but-not-yet-drained proc does not set `_user_stopped` or signal a dead pid.
2. Defense-in-depth: in `final_status_text` move the `rc == 2` honest branch ABOVE the `user_stopped` branch — rc=2 is always pre-exec with no resumable state and must never be relabeled as a resumable stop. Add a test for `final_status_text(2, ..., user_stopped=True)` staying '재개 불가'.

## Notes
Low severity (≤120ms race) but a genuine correctness regression from QA-054. Verified by executing the pure function. Not covered by QA-032/047/054 (none exercise a stale flag on an exited non-stop process).
