# QA-015 — CIDR/range cap is checked AFTER full materialization, so a large network (IPv4 /8 or any IPv6 CIDR) hangs/OOMs before the cap can fire

- **group**: I    **disposition**: FIX    **severity**: high    **dimension**: portability-edge
- **dedup_key**: `expand_targets_cidr_cap_after_materialization`
- **location**: scanner/scanops_scanner.py:205-227 (expand_targets); reached unconditionally from create_plan at line 545

## Evidence
```
Line 217 `hosts.extend(str(ip) for ip in net)` fully materializes the entire network into a Python list, and only THEN line 225 `if len(hosts) > cap:` raises. The cap check is per-target, applied after each target is fully expanded, so it cannot interrupt expansion WITHIN one oversized target. Verified: `expand_targets(['10.0.0.0/8'], cap=1000)` (16.7M hosts) did not return within a 10s timeout; `expand_targets(['2001:db8::/96'], cap=10)` (4.29e9 hosts) hung past 8s; `expand_targets(['2001:db8::/64'], ...)` would attempt 1.8e19 addresses. create_plan calls `expand_targets(raw_targets, args.max_hosts)` (line 545) unconditionally even in non-batched mode, so a single mistyped /8 or any IPv6 CIDR freezes the process / exhausts memory before any scan starts.
```
## Impact
The --max-hosts safety valve is defeated by exactly the input it exists to guard against. A typo like `10.0.0.0/8` instead of `/24`, or any IPv6 CIDR, makes the scanner hang or get OOM-killed before printing anything — looks like a freeze to the operator, with no error and no scan.

## Proposed fix
Before materializing, check `net.num_addresses > cap` (cheap, no expansion) and raise the same ValueError. When expanding, count incrementally and break/raise as soon as `len(hosts) > cap` is reached mid-network (e.g. iterate with a running counter or `itertools.islice(net, cap+1)`), instead of `hosts.extend(... for ip in net)`.

## Verifier rationale
Confirmed from scanner/scanops_scanner.py. expand_targets (lines 205-227) materializes an entire CIDR via `hosts.extend(str(ip) for ip in net)` at line 217 and only checks the cap at line 225 (`if len(hosts) > cap`). The check is per-target, applied AFTER each target is fully expanded, so it cannot interrupt expansion within a single oversized CIDR. create_plan calls `expand_targets(raw_targets, args.max_hosts)` at line 545 unconditionally, BEFORE the `args.batch_size > 0` gate at line 546, so the materialization happens even in default non-batched mode (batch_size default 0, max_hosts default 65536). Empirically reproduced: `expand_targets(['10.0.0.0/8'], 1000)` (16.7M hosts) hangs past an 8s timeout (process exit 124) instead of immediately raising the cap ValueError; ipaddress confirms /8 = 16,777,216 addresses and v6 /96 = 4.29e9. The IPv4 RANGE_RE path (line 218-222) is safe because line 220 caps `hi > 255`, but the CIDR path has no size guard. Net effect: the --max-hosts safety valve is defeated by exactly the oversized input it exists to guard against; a typo like /8 or any large IPv6 CIDR freezes/OOMs the process with no output before any scan starts. This is distinct from ISSUE-001 (UDP stage failure aborting the plan). Severity high rather than critical: it requires operator misinput (an oversized CIDR), is recoverable by killing the process, and corrupts no results — but it is a real, silent denial-of-service of the tool.
