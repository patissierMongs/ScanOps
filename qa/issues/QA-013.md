# QA-013 — protocol_ports misclassifies a bare port that follows a U: token as UDP, silently dropping TCP ports and scanning them as UDP

- **group**: H    **disposition**: FIX    **severity**: medium    **dimension**: result-correctness
- **dedup_key**: `protocol_ports_sticky_proto_silently_drops_tcp_stages`
- **location**: scanner/scanops_scanner.py:327-347 (protocol_ports), called by auto_tcp_discovery_ports:350-357 and auto_udp_ports:360-365

## Evidence
```
protocol_ports keeps a sticky `current` protocol from the last `T:`/`U:` prefix. With `--ports U:53,80`: `protocol_ports('U:53,80','T') -> []` and `protocol_ports('U:53,80','U') -> ['53','80']` (verified). So auto_tcp_discovery_ports('U:53,80') -> '' and auto_udp_ports('U:53,80') -> 'U:53,80'. In execute_auto, auto_tcp_discovery_ports()=='' is falsy (713) so BOTH tcp_discovery and tcp_identify are skipped (732-733), and port 80 is scanned as a UDP port.
```
## Impact
A user who specifies mixed ports as `U:53,80` (intending 80 as TCP, a natural reading) gets: TCP scanning entirely skipped, and TCP port 80 probed over UDP where it will never respond. The user's TCP intent is silently lost with no error. nmap's own `-p` semantics also make a bare port after `U:` belong to UDP, so this is subtle, but the tool gives no warning that TCP ports were dropped.

## Proposed fix
Reject or warn when ports_override contains a U:/T: prefix followed by bare (unprefixed) ports, or normalize by requiring every segment to carry its protocol prefix in mixed specs. At minimum, when auto_tcp_discovery_ports() returns '' because the override had no T-classified ports BUT the override is non-empty and contains digits, raise a clear error instead of silently skipping all TCP stages.

## Verifier rationale
Confirmed by reading and executing the code. protocol_ports (scanner/scanops_scanner.py:327-347) carries a sticky `current` protocol from the last T:/U: prefix, so a bare port following a U: token is classified UDP. Verified empirically: protocol_ports('U:53,80','T')==[] and ('U:53,80','U')==['53','80']; auto_tcp_discovery_ports({'ports_override':'U:53,80'})=='' and auto_udp_ports==' U:53,80'. In execute_auto (713) the empty TCP spec is falsy, so the else at 731-735 appends tcp_discovery AND tcp_identify as SKIPPED, and port 80 is then scanned over UDP via auto_udp_ports. validate_ports/PORTS_RE (97,180-186) accept 'U:53,80' without complaint, and the only protocol_ports guard (551) fires solely under --tcp-only, so in the default auto workflow there is NO warning that all TCP scanning was dropped. The defect is real and reachable (--ports is free text in both CLI and GUI). I downgrade from high to medium for two reasons: (1) the classification is actually consistent with nmap's own -p semantics (bare port after U: is UDP), which the finding concedes, so it is not a misclassification vs nmap; (2) it requires a specific, atypical mixed-spec form. The genuinely defensible defect is the SILENT loss of all TCP coverage with no error in a security-exposure scanner. Surgical fix: when ports_override is non-empty but auto_tcp_discovery_ports() yields no TCP ports in the auto workflow, warn or error (generalize the existing line-551 guard beyond --tcp-only) rather than silently skipping both TCP stages.
