# QA-028 — GUI couples pure CLI-output logic to a top-level tkinter import, making it untestable / unimportable on headless hosts

- **group**: R2    **disposition**: FIX    **severity**: medium    **dimension**: ui-ux-testability
- **dedup_key**: `gui-toplevel-tkinter-import-blocks-pure-logic`
- **location**: scanner/scanops_scanner_gui.py:13 (`from tkinter import BooleanVar, StringVar, Tk, filedialog, messagebox`), :14 (`from tkinter import scrolledtext, ttk`); affects pure function `parse_marker` (:20) and the GUI↔CLI marker contract test `test_gui_parse_marker_handles_both_resume_line_forms` (backend/tests/test_standalone_scanner.py:900)

## Evidence
```
$ python3 -c "import tkinter"
ModuleNotFoundError: No module named 'tkinter'

$ pytest backend/tests/test_standalone_scanner.py::test_gui_parse_marker_handles_both_resume_line_forms
...
>   from tkinter import BooleanVar, StringVar, Tk, filedialog, messagebox
E   ModuleNotFoundError: No module named 'tkinter'
scanner/scanops_scanner_gui.py:13: ModuleNotFoundError
```
`parse_marker` is a Tk-free pure function (its own docstring says "순수 함수, Tk 불필요 → 단위 테스트 가능"), and it is the contract that drives the GUI's resume-autofill / partial / warning state from CLI stdout. But the test must `exec_module` the whole GUI file, which fails at the top-level `from tkinter import ...` on any host without a display/Tk (headless CI, minimal Python, this container). So the one piece of GUI logic explicitly designed to be unit-testable cannot be imported at all.

## Impact
The GUI↔CLI marker contract — the thing that makes the GUI surface "재개 실행" autofill, partial completion, and warning counts — has zero test coverage on any headless environment. A wording drift in the CLI's `resume with:` / `partial:` / `warning:` lines would silently break the GUI and no CI without Tk could catch it. This is a verification-reliability hole for the UI, not a cosmetic one.

## Proposed fix
Guard the tkinter imports so the module is importable for its pure logic without a display: wrap `from tkinter import ...` in `try/except ImportError`, binding the names to `None` on failure, and raise a clear, user-facing error only inside `main()` (when the GUI actually needs to render). Pure functions (`parse_marker`) and module-level constants (RUN_MODE_*, PORT_PRESETS, SCAN_TYPES) then import everywhere. Alternative: extract `parse_marker` + constants into a tkinter-free helper module imported by both the GUI and the test.

## Verifier rationale
Confirmed by direct repro in this environment. The coupling is real and the affected logic is explicitly documented as Tk-independent, so guarding the import is a faithful fix (not a behavior change for real GUI use — running the GUI still requires Tk, which is correct). Not overengineering: minimal try/except, no new module required, restores the intended testability.
