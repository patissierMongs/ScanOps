# QA-006 — fail_plan writes manifest with returncode==0 filter, so a mid-plan failure yields a manifest that omits the very stage data already on disk

- **group**: A    **disposition**: FIX    **severity**: medium    **dimension**: failure-isolation
- **dedup_key**: `fail_plan-incomplete-failure-bundle-manifest-and-zip`
- **location**: scanner/scanops_scanner.py:692-697 (fail_plan) calling write_manifest at :696

## Evidence
```
def fail_plan(plan, state_path, rc):
    plan["status"] = "failed"
    plan["finished_at"] = now_iso()
    write_json(state_path, plan)
    write_manifest(plan)   # :696 — manifest still applies returncode==0 filter at :627/:634
    return rc
```
## Impact
On failure, fail_plan DOES emit a manifest (good intent), but write_manifest's returncode==0 + skipped filter means the failing stage's partial XML is excluded, and (when udp/identify failed) the manifest may be near-empty even though tcp_discovery/tcp_identify XML exist on disk. Worse, fail_plan never zips (zip only happens in finish_plan), so an operator who passed --zip gets no zip at all on partial failure and must hand-collect files. The 'failed' manifest is thus a misleading record of what was actually captured.

## Proposed fix
Have fail_plan (or a unified finalizer) include all parseable XML produced so far and honor --zip even on partial completion, so a failed/partial run still ships a usable bundle. Mark partial entries explicitly rather than dropping them.

## Verifier rationale
Verified in scanner/scanops_scanner.py. fail_plan (lines 692-697) DOES emit a manifest via write_manifest(plan) at :696 (no zip arg). write_manifest filters all_xml_files and import_xml_files to runs with returncode==0 and not skipped (lines 627 and 634). run_nmap_stage records the failing run with returncode=rc (non-zero, :682) and files=existing_outputs(base) (:685, which globs whatever .xml nmap wrote before exiting, see existing_outputs :433-439). Therefore the failing stage's partial XML IS on disk but IS excluded from the manifest by the returncode==0 filter. CONFIRMED. Also confirmed: zipping only happens in finish_plan (create_zip called at :660-661); fail_plan takes no zip_outputs param and never zips, so an operator who passed --zip gets no bundle on partial failure and must hand-collect files. CONFIRMED.

One material correction to the finding's framing: the title ("omits the very stage data already on disk") and the "near-empty manifest" claim are OVERSTATED. Successful prior stages (tcp_discovery, tcp_identify) are recorded as returncode==0 runs and ARE included in the manifest; only the single FAILING stage's partial XML is dropped. So when udp_identify fails, the failed manifest still lists the tcp_discovery/tcp_identify XML (import_xml_files even still lists tcp_identify) — it is not near-empty, just missing the aborted stage. The real, defensible defects are: (1) the failing stage's partial XML is silently dropped from the manifest, and (2) --zip is entirely ignored on partial failure.
