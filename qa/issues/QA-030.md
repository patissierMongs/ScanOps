# QA-030 — Single-workflow scan summary always reports live_hosts=0 even when hosts are up with open ports (misleading result)

- **group**: R2    **disposition**: FIX    **severity**: medium    **dimension**: result-reliability
- **dedup_key**: `single-workflow-summary-live-hosts-always-zero`
- **location**: scanner/scanops_scanner.py:837 `scan_findings` — `live` is only populated in the `stage == "tcp_discovery"` branch (:850); single-workflow runs have `stage_id == ""` and fall into the `else` branch (:854) which updates tcp/udp ports but never `live`. Printed by `print_scan_summary` (:898).

## Evidence
```
# host up + 443 open, single-workflow run (stage_id="")
scan_findings(...) -> {'live_hosts': 0, 'open_tcp': 1, 'open_udp': 0, 'importable': 1}
live_hosts_from_xml(<same xml>) -> ['100.64.0.5']
```
For any `--workflow single` profile (basic/quick/light/phase1) — the GUI's "단일 실행" modes — the end-of-scan line prints `summary: live_hosts=0 open_tcp=N ...` whenever ports are found, because live hosts are only counted from a `tcp_discovery` stage that single workflow never runs.

## Impact
The user sees a self-contradicting summary: "0 live hosts" next to "N open TCP ports". For the GUI's quick/precision single-run modes this is the normal, expected output, so the headline live-host count is wrong on every single-workflow scan. Erodes trust in the result and can be misread as "nothing was found / host is down" when services were in fact identified.

## Proposed fix
In `scan_findings`, also count a host as live when it has >=1 open port in any identify/single-stage XML (a host with an open port is unambiguously up), unioned with the existing discovery up-hosts. Add a helper `hosts_with_open_ports_from_xml(path)` and update the `udp_identify` and `else` (tcp_identify/single) branches to `live.update(...)`. This fixes single-workflow counts and additionally lets UDP-only live hosts register, without inflating from `-Pn` (which marks dead hosts up): only hosts with a real open port are added. Auto-workflow counts are unchanged (identify hosts ⊆ discovery up-hosts; it's a set union). Add a regression test asserting single-workflow summary live_hosts>0 when a port is open.

## Verifier rationale
Confirmed by direct reproduction against the live `scan_findings` function. The undercount is structural (live only ever set in the discovery branch), affects every single-workflow run, and the proposed union-with-open-port-hosts definition is correct and non-inflating. Distinct from QA-012 (which is about honestly surfacing *empty* results); this is about *non-empty* results showing a wrong live count. Moderate, surgical fix — not overengineering.
