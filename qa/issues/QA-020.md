# QA-020 — Standalone scanner enforces no SCANOPS_SCAN_SCOPE allowlist — a copy-and-run footgun that can scan out-of-org networks

- **group**: I    **disposition**: FIX    **severity**: high    **dimension**: ux-gaps
- **dedup_key**: `scanner-no-scan-scope-allowlist`
- **location**: scanner/scanops_scanner.py:163-177 (collect_targets/validate_targets) and create_plan:534-583; contrast backend/scanops/scanning/scope.py:57-93 (check_scope/check_raw_scope)

## Evidence
```
validate_targets only checks a character regex: `bad = [t for t in targets if not TARGET_RE.match(t)]` (line 175). There is zero reference to SCANOPS_SCAN_SCOPE anywhere in scanner/ (grep -i scope returns no matches in scanner/). The server, by contrast, gates every scan via check_scope(): `if not nets: return ... bad = [h for h in hosts if not _in_scope(h, nets)] ... raise ValueError("허용된 스캔 대역(scope) 밖의 대상입니다...")`. README.md:57-59 documents SCANOPS_SCAN_SCOPE as the guard against "오타·잘못 붙여넣은 사외 대역 스캔 사고" and explicitly tells operators to copy scanner/scanops_scanner.py to a scanner host (README.md:40, line 5-6 of the module docstring).
```
## Impact
The scanner is documented and designed to be copied standalone onto arbitrary hosts and run by auditors. A single typo (10.0.0.0/8 -> 100.0.0.0/8) or a pasted external CIDR causes an active SYN/UDP scan (AUTO_TCP_DISCOVERY_FLAGS uses -sS -p T:1-65535 -T4 --max-parallelism 100) against an out-of-scope / third-party network with no guard whatsoever. The exact accident the server's scope gate was built to prevent is wide open in the standalone tool, and operators who trust the README's scope guarantee will assume they are protected when they are not.

## Proposed fix
Port the scope logic: read an env var (e.g. SCANOPS_SCAN_SCOPE) in create_plan after collect_targets, and if set, expand targets and reject any host outside the allowed CIDRs (mirroring backend/scanops/scanning/scope.py:check_scope). At minimum honor it in the GUI too. Document in scanner/README that scope is NOT enforced unless the env var is set, so operators do not assume server-equivalent safety.

## Verifier rationale
Verified against source. validate_targets (scanner/scanops_scanner.py:174-177) only does a character-class check: `bad = [t for t in targets if not TARGET_RE.match(t)]`, where TARGET_RE (line 96) = `^[A-Za-z0-9_.:/\-]+$`. This accepts any IP/CIDR/hostname-shaped token, so a typo like 100.0.0.0/8 or a pasted external CIDR passes. create_plan (534-583) and collect_targets (163-171) contain no allowlist logic; create_plan only does collect_targets -> expand_targets (a --max-hosts cap check, not scope). Grep for scope|SCAN_SCOPE|SCANOPS_SCAN_SCOPE across all of scanner/ returns zero matches (the only env access is the GUI copying os.environ for encoding at scanops_scanner_gui.py:354). By contrast the server gates every scan via backend/scanops/scanning/scope.py:57-69 check_scope (raises ValueError on out-of-scope hosts), and that module's docstring (lines 3-6) cites the exact 10.0->100.0 typo scenario. The standalone tool is explicitly documented/designed to be copied and run independently (module docstring scanner/scanops_scanner.py:4-6; README.md:40).
