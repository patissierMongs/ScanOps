# QA-039 — Summary open_tcp/open_udp count DISTINCT port numbers across all hosts, under-reporting exposure on multi-host scans

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: result-reliability
- **dedup_key**: `summary-counts-distinct-port-numbers-not-host-port-pairs`
- **location**: scanner/scanops_scanner.py scan_findings (`tcp: set[int]` / `udp: set[int]`, reports `len(tcp)`/`len(udp)`)

## Evidence
`scan_findings` accumulates bare port NUMBERS (`open_ports_from_xml` returns ints with no host association), so the same port open on N hosts collapses to one. Verified: a two-host XML (10.0.0.1: 22+443, 10.0.0.2: 443) yields `open_tcp=2`, not 3 (host,port) pairs. For a /24 where 50 hosts each expose 443+3389, the operator-facing line reports `open_tcp=2`, reading as "only 2 open TCP ports on the whole network" — a severe under-representation of attack surface. The docstring labels it "열린 TCP/UDP 포트 수" (a total).

## Proposed fix
Count `(host, port)` pairs per protocol: add `open_host_ports_from_xml(path, proto)` returning `(ip, portid)` and accumulate those into the sets. Scan/import data is unaffected; only the headline count is corrected.
