# QA-010 — --scan-type connect still emits -sU in auto UDP stage, so the 'no privileges' choice fatally fails the UDP identify

- **group**: E    **disposition**: FIX    **severity**: high    **dimension**: nmap-coverage
- **dedup_key**: `connect-scan-type-still-emits-sU-in-auto-udp-stage`
- **location**: scanner/scanops_scanner.py:267-272 (set_scan_type), :103 (SCAN_TYPE_FLAGS), :72-76 (AUTO_UDP_IDENTIFY_FLAGS); GUI scanner/scanops_scanner_gui.py:48-52 (SCAN_TYPES label '권한 불필요 - TCP Connect')

## Evidence
```
SCAN_TYPE_FLAGS = {"-sS", "-sT"} and set_scan_type does `flags = [f for f in flags if f not in SCAN_TYPE_FLAGS]; return [mapped, *flags]`. It only ever strips/replaces -sS/-sT and never touches -sU. AUTO_UDP_IDENTIFY_FLAGS begins with -sU. Dry-run with --scan-type connect produces: `-sT -sU -Pn -n -sV ... -p U:7,53,...`. -sU always requires raw-socket (root/admin) privileges; it cannot be performed via connect().
```
## Impact
A user who explicitly picks 'TCP Connect / no admin privileges required' in the GUI still gets a UDP identify stage that requires admin. On an unprivileged host nmap aborts the UDP stage with 'requires root privileges', which (per ISSUE-001's fail_plan path at :748) marks the whole plan failed. The user's deliberate privilege-free choice is silently violated and the scan dies.

## Proposed fix
When scan_type=='connect' (no raw-socket privilege available), the UDP identify stage must be skipped with an explicit append_skipped_stage reason instead of emitting -sU. Either strip -sU and skip udp_identify when scan_type is connect, or surface an upfront error/warning that UDP scanning is unavailable without privileges.

## Verifier rationale
Confirmed from code and empirical dry-run. set_scan_type (scanner/scanops_scanner.py:267-272) strips only SCAN_TYPE_FLAGS = {"-sS","-sT"} (:103) and never removes -sU. AUTO_UDP_IDENTIFY_FLAGS (:72-76) starts with -sU. build_auto_flags(udp_identify) (:398) builds from those flags, then apply_auto_modifiers (:369) calls set_scan_type with plan scan_type. Running build_auto_flags({'scan_type':'connect'}, 'udp_identify') yields: -sT -sU -Pn -n -sV --open --reason -T4 --max-retries 2 --script ... -p U:7,53,... — i.e. -sU is still present alongside -sT. The GUI maps "권한 불필요 - TCP Connect" -> connect (gui :48-52) and passes --scan-type connect (gui :310-312), so this is reachable end-to-end. -sU requires raw-socket (root/admin) privileges and cannot run via connect(); on an unprivileged host nmap exits non-zero, which hits the fail_plan path for udp_identify (:748-749, :757-758, :762-763), aborting the whole plan and discarding successful TCP results. This is distinct from ISSUE-001: ISSUE-001 is the generic "UDP exit!=0 -> fail_plan" cascade; this finding is that the user's explicit no-privileges choice is silently violated by still emitting -sU, which deterministically guarantees that failure for exactly the users who opted out of needing admin. Proposed fix is sound: when scan_type=='connect', skip udp_identify with append_skipped_stage (mirroring the existing tcp_only skip at :737-739) rather than emitting an -sU command that cannot succeed.
