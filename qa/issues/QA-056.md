# QA-056 — xml_has_hosts treats a content-less identify <host> (MAC-only, or status=down/all-filtered) as importable, re-opening QA-049: it suppresses the discovery fallback and drops the real open-port data

- **group**: R5   **disposition**: FIX   **severity**: high   **dimension**: result-reliability
- **dedup_key**: `xml-has-hosts-too-weak-contentless-host-suppresses-fallback`
- **location**: scanner/scanops_scanner.py importable_xml (uses xml_has_hosts) vs the QA-050 helpers open_host_ports_from_xml / hosts_with_open_ports_from_xml (skip MAC-only/empty-ip, require an open port)

## Evidence
QA-049 filters importable_xml's non-discovery XMLs through `xml_has_hosts(p)`, but `xml_has_hosts` only checks `root.find('.//host') is not None` — mere element existence, not importable content. The QA-050 counting helpers are stricter. So an rc=0 `tcp_identify` XML whose only `<host>` is MAC-only, OR has an ipv4 address but status=down with all ports filtered/closed (a host that went unreachable between discovery and -Pn identify — common), passes `xml_has_hosts=True`, occupies `seen`, and short-circuits the QA-038 discovery fallback. Reproduced: MAC-only identify host + discovery(10.0.0.1: 22,443 open) → `import_xml_files=['...tcp_identify.xml']` only, discovery dropped, while summary still reports live_hosts=1 open_tcp=2. A downstream import gets a host-less/port-less XML on a 'done'/'partial' scan — the exact QA-049 failure mode via a new trigger its test (truly element-less XML) does not catch.

## Proposed fix
Make the importability predicate consistent with the QA-050 helpers: add `xml_has_usable_host(p)` = `bool(live_hosts_from_xml(p) or hosts_with_open_ports_from_xml(p))` (a host with a non-MAC address that is status=up or has an open port) and use it in importable_xml's two passes instead of `xml_has_hosts`. Add a test with an rc=0 identify XML whose sole host is MAC-only / status=down-all-filtered, asserting the discovery fallback still fires.
