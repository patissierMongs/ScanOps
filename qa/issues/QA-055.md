# QA-055 — rc<0 stop status promises a resumable state.json after a force-killed dry-run (which writes no state)

- **group**: R5   **disposition**: FILE-ONLY (near-unreachable race; recorded only)   **severity**: low   **dimension**: ui-ux
- **dedup_key**: `qa047-stop-message-promises-state-after-dryrun`
- **location**: scanner/scanops_scanner_gui.py final_status_text rc<0 branch, reached from a dry-run (`명령 확인`) force-kill

## Evidence
A dry-run runs the CLI with `--dry-run`; `execute()` returns before `out_dir.mkdir`/any `write_json`, so no `state.json` is created. If the user clicks 중지 during a (POSIX) dry-run and the 6s force-kill fires SIGKILL, `final_status_text(-9, ...)` returns "중지됨(신호 9) — 'state.json 선택' 후 [재개 실행]" — pointing at a state file that does not exist.

## Disposition note
The adversarial verifier rated this **file-only**: a dry-run is a millisecond-scale `print_plan` with no subprocess work, so the 6s force-kill race is effectively unreachable, and clicking resume on a missing state simply errors (no data loss). Recorded for completeness. If the QA-054 `_user_stopped` refactor lands, threading a dry-run flag into the stop message is a trivial optional follow-on, but not warranted on its own.
