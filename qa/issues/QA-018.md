# QA-018 — Duplicate / overlapping targets are never deduplicated, causing redundant scans and output-file collisions in batched mode

- **group**: I    **disposition**: FIX    **severity**: low    **dimension**: portability-edge
- **dedup_key**: `targets-not-deduplicated-before-batching`
- **location**: scanner/scanops_scanner.py:205-227 (expand_targets), 163-171 (collect_targets), 230-233 (make_batches), 404-411 (output_base)

## Evidence
```
Neither collect_targets nor expand_targets deduplicates. Verified: `expand_targets(['10.0.0.1','10.0.0.1'])` returns the dup, and `expand_targets(['10.0.0.0/30','10.0.0.1'])` returns `['10.0.0.0','10.0.0.1','10.0.0.2','10.0.0.3','10.0.0.1']` (10.0.0.1 twice). In batched mode (batch_size==1) each duplicate becomes its own batch; output_base uses `target_label` of the batch (line 408) so two batches resolving to the same host produce filenames distinguished only by the batch index, but the duplicate host is scanned twice. In non-batched mode the duplicates are all passed to a single nmap invocation.
```
## Impact
Operators who paste overlapping CIDRs plus explicit hosts (common when assembling internal scopes) silently double-scan hosts — wasted time and, with batching, redundant output files for the same IP. Not corrupting, but inefficient and confusing in manifests.

## Proposed fix
Deduplicate while preserving order after expansion (e.g. dict.fromkeys) in expand_targets or collect_targets, before make_batches.

## Verifier rationale
Confirmed by reading the code and executing the functions. No deduplication exists anywhere in the target pipeline: collect_targets (scanner/scanops_scanner.py:163-171) only strips/validates; expand_targets (205-227) only expands CIDR/ranges and appends; make_batches (230-233) just slices. The only set/seen usage (lines 450, 473-485) is for parsing scan results, not targets.

Empirically verified (ran the real module):
- expand_targets(['10.0.0.1','10.0.0.1']) -> ['10.0.0.1','10.0.0.1'] (dup kept)
- expand_targets(['10.0.0.0/30','10.0.0.1']) -> ['10.0.0.0','10.0.0.1','10.0.0.2','10.0.0.3','10.0.0.1'] (10.0.0.1 twice)
- make_batches(expanded, 1) -> [['10.0.0.0'],['10.0.0.1'],['10.0.0.2'],['10.0.0.3'],['10.0.0.1']] (duplicate host gets its own batch index 4)
- target_label of batch 1 and batch 4 both == '10.0.0.1'

So in batched mode the duplicate host IS scanned a second time (extra batch -> extra nmap invocation + redundant output files + redundant manifest entry). One precision note: the title's 'output-file collisions' is slightly overstated — output_base (line 411) appends .b{index:04d}, so batch 1 -> ...10.0.0.1.b0001... and batch 4 -> ...10.0.0.1.b0004..., distinct filenames, NOT a collision. The finding body itself acknowledges this correctly ('distinguished only by the batch index, but the duplicate host is scanned twice'), so the body is accurate even though the headline word 'collision' is loose.
