# QA-022 — Resume UX: the only resume hint goes to stderr from the CLI; the GUI gives no resume affordance after a failed/stopped scan and never auto-fills the state path

- **group**: G    **disposition**: FIX    **severity**: medium    **dimension**: ux-gaps
- **dedup_key**: `resume-hint-missing-on-failure-cli-and-gui`
- **location**: scanner/scanops_scanner.py:791 (stderr resume hint) and fail_plan:692-697 (no hint at all); GUI resume section scanner/scanops_scanner_gui.py:171-176, 259-262

## Evidence
```
On a stage failure the CLI calls fail_plan (scanner lines 692-697) which writes status "failed" and returns rc but prints NO resume guidance — only the KeyboardInterrupt path prints the `--resume` hint (line 791). In the GUI, resume requires the user to manually browse for `*.state.json` via _browse_resume (gui line 259) and click 재개 실행; nothing auto-populates self.resume_path after a run fails, and the GUI never echoes the state-file location (the CLI prints it only in the interrupt case, to stderr which is merged into the log).
```
## Impact
After the documented common failure (e.g. ISSUE-001's UDP-stage nonzero exit aborting the plan), the user sees `종료 코드 7` in the GUI with zero indication that the partial scan is resumable or where the state file is. The resume feature exists but is effectively undiscoverable at the moment it is needed, so users re-run the whole expensive full-port scan from scratch.

## Proposed fix
Have fail_plan print the same `Resume with: --resume <state_path>` hint the interrupt path prints. In the GUI, on a nonzero/failed exit, auto-set self.resume_path to the just-run state.json (derivable from output_dir + name) and surface a one-click '재개' prompt.

## Verifier rationale
Confirmed against the actual code on every claim. CLI: fail_plan (scanner/scanops_scanner.py:692-697) writes status "failed" + manifest and returns rc but prints NOTHING — no resume hint, no state path. The only place a resume hint is emitted is the KeyboardInterrupt branch at line 791 (`"\ninterrupted. Resume with: --resume " + str(state_path)`, to sys.stderr). I also verified the state path is never printed on any normal path: print_plan (601-617) prints only `output:`/`batches:`; finish_plan (662) prints `done: <manifest_path>`, not the state file. So after a stage failure (e.g. ISSUE-001's UDP nonzero exit), the user has no on-screen indication the partial scan is resumable or where the .state.json lives. GUI: resume is manual-only — _browse_resume (scanner/scanops_scanner_gui.py:259-262) + the 재개 실행 button (line 176). _drain_output's "done" handler (lines 398-403) on nonzero rc only sets status `종료 코드 {rc}` and logs `exit code: {rc}`; it never sets self.resume_path and never surfaces the state-file location. The GUI never derives the state path itself and only echoes CLI stdout/merged-stderr (stderr=STDOUT at line 359), which on failure contains no path. Net effect: the resume feature is undiscoverable exactly when it is needed, so users re-run the whole expensive full-port scan. This is a genuine UX defect that compounds the already-filed reliability issue. Severity medium: no data loss or crash (partial results ARE saved and resumable), but it defeats the value of an existing recovery feature. Worth fixing: the fix is surgical and grounded — add the same resume hint
