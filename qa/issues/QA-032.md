# QA-032 — GUI promises "[재개 실행] 가능" for every non-zero/non-130 exit, including rc=2 validation errors where no state.json exists

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: ui-ux
- **dedup_key**: `final-status-false-resume-promise-on-rc2`
- **location**: scanner/scanops_scanner_gui.py (_final_status: returns "실패(종료 코드 {rc}) — [재개 실행] 가능" for any rc not in {0,130})

## Evidence
`_final_status` advertises resume for ALL rc not in {0,130}. But rc=2 means `create_plan` raised a ValueError (bad ports/targets/IPv6/scope/CIDR-cap/nmap-missing) BEFORE `execute()` ran, so no state file was written and `_resume_hint` is empty. The autofill (`_drain_output`) IS correctly gated on `_resume_hint`, so the two halves of the same handler disagree: the path Entry stays empty while the status text tells the operator to click 재개 실행.

## Proposed fix
Gate the resume text on `self._resume_hint or rc == 130`. For rc==2 surface a distinct "입력/설정 오류 — 재개 불가" message; for other rc with no hint say "재개할 상태가 없습니다". (Extract a pure `final_status_text(rc, partial, warn_count, has_resume)` helper so it is unit-testable headless.)

Not covered by QA-022 (introduced the affordance, never considered no-state exits), QA-026 (rc=2 visual distinctness), or QA-027 (inverse: non-zero hiding partial results). Verified in code.
