# QA-033 — Resume-path autofill never refreshes: a second failed/interrupted scan resumes the FIRST scan's state

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: ui-ux
- **dedup_key**: `resume-path-autofill-stale-across-runs`
- **location**: scanner/scanops_scanner_gui.py (_drain_output 'done' handler: `if self._resume_hint and not self.resume_path.get().strip(): self.resume_path.set(...)`)

## Evidence
The `not ...get().strip()` guard writes the field only when empty. After scan A fails, resume_path = A.state.json. When scan B (different targets/output) then fails, B's hint is parsed into `_resume_hint` but autofill is skipped because the field is non-empty — it still points at A. Clicking 재개 실행 resumes scan A, not B, with no warning. resume_path is also never cleared after a subsequent successful run. Per-run state paths genuinely differ (deterministic from output_dir+name+target), so this silently resumes the wrong state file.

## Proposed fix
On each 'done' with a hint, overwrite resume_path with the latest `_resume_hint`. Clear resume_path at the start of a fresh (non-resume, non-dry-run) `_start` so a new run cannot inherit a previous run's state file.

## Notes
QA-022 introduced the autofill (absence fix) and did not consider multi-run staleness; QA-023/009 are CLI/engine guards. Verified in code; surgical fix.
