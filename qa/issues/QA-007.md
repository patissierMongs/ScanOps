# QA-007 — No nmap --host-timeout and no subprocess timeout: a single host can hang the whole scan forever

- **group**: B    **disposition**: FIX    **severity**: high    **dimension**: stability-hang-stop
- **dedup_key**: `no-host-timeout-unbounded-scan-hang`
- **location**: scanner/scanops_scanner.py:674 (run_nmap_stage: subprocess.call(cmd, shell=False)); flag sets AUTO_TCP_DISCOVERY_FLAGS:54-59, AUTO_TCP_IDENTIFY_FLAGS:64-67, AUTO_UDP_IDENTIFY_FLAGS:72-76

## Evidence
```
rc = subprocess.call(cmd, shell=False)  # line 674, no timeout= argument. None of the AUTO_* flag lists contain --host-timeout (grep for host-timeout returns only the comment at line 68). A full-port SYN scan uses -p T:1-65535 with -T4 --max-retries 2 but nothing bounds per-host time. UDP identify deliberately forbids --max-scan-delay (comment line 68), so a heavily rate-limited / filtered UDP host backs off indefinitely.
```
## Impact
Against a firewalled / silently-dropping internal host (very common for exposure scanning), nmap can run for hours on a single target with no upper bound. subprocess.call() has no timeout, so the scanner process blocks indefinitely. The whole plan stalls on one bad host; the user's only recourse is to hard-kill, which then corrupts the state (see other findings). Production scans become unpredictable in wall-clock time.

## Proposed fix
Add --host-timeout (e.g. 30m configurable) to each AUTO_* flag set so nmap self-bounds per host and exits 0 with partial results; nmap treats host-timeout as non-fatal. Optionally also pass a generous timeout= to subprocess.call as a hard backstop and treat TimeoutExpired by killing the child and recording the stage as timed-out rather than hanging.

## Verifier rationale
Confirmed from the actual code. (1) scanner/scanops_scanner.py:674 `rc = subprocess.call(cmd, shell=False)` has no timeout= argument; this is the CLI engine's only nmap invocation point and execute_single (line 700-708) blocks on it synchronously with no watchdog. (2) grep for "host-timeout" returns zero hits in both files — the only timeout flag present is `--script-timeout 10s` (lines 66, 75), which bounds individual NSE scripts, NOT overall per-host scan time; a host whose ports are all silently dropped never reaches script execution, so that flag does not help. (3) AUTO_TCP_DISCOVERY_FLAGS (lines 54-59) does a full-port `-p T:1-65535` SYN scan with `-T4 --max-retries 2`; --max-retries bounds retransmits per probe but not total host time, so a filtered/dropping host can take hours. (4) The comment at line 68 confirms `--max-scan-delay` is deliberately forbidden for UDP identify, so a rate-limited UDP host's scan delay backs off without bound, exactly as claimed. The result: against a firewalled/silently-dropping internal host (very common in exposure scanning) the scanner can hang indefinitely on one target with no upper bound and no auto-recovery. The GUI offers a manual taskkill via _stop (scanner_gui.py:380-390) using Popen, but (a) the CLI itself has zero recourse and (b) hard-kill is the only escape even in the GUI. This is genuinely distinct from ISSUE-001 (which is about exit!=0 aborting the plan) — here the scan never returns at all. In scope for stability/reliability of a production internal scanner. Worth a surgical fix: add `--host-timeout` to each AUTO_* flag
