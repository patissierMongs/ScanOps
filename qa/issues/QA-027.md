# QA-027 — GUI surfaces non-zero CLI exit purely as a numeric code with no indication that partial results exist

- **group**: D    **disposition**: FIX    **severity**: low    **dimension**: failure-isolation
- **dedup_key**: `gui-nonzero-exit-no-partial-results-hint`
- **location**: scanner/scanops_scanner_gui.py:398-403 (_drain_output done handling)

## Evidence
```
rc = int(payload)
... self.status.set("완료" if rc == 0 else f"종료 코드 {rc}")
self._append_log(f"\nexit code: {rc}\n")
```
## Impact
Because the CLI escalates any stage failure to a non-zero process exit (fail_plan -> main returns rc), the GUI shows only 'exit code 7' (or 1/2) with no hint that a manifest/zip with usable discovery/identify XML may still have been written. An operator sees 'failure', assumes nothing was captured, and discards a run that actually produced exploitable exposure data on disk. The binary fail-the-plan model in the CLI thus directly degrades the GUI's reliability story.

## Proposed fix
Once the CLI distinguishes partial from fatal (findings above), have it print/emit a 'partial: manifest written' line on recoverable failure and a distinct exit code; GUI should map that to a 'completed with warnings — results saved' status pointing at the output dir, rather than a bare error code.
