# QA-008 — Corrupt/truncated discovery XML is swallowed, yielding an empty-but-"done" scan with exit 0 (a host with open ports is reported as zero exposure)

- **group**: C    **disposition**: FIX    **severity**: high    **dimension**: result-correctness
- **dedup_key**: `discovery_xml_parseerror_swallowed_silent_empty_done`
- **location**: scanner/scanops_scanner.py:445-448 (open_ports_from_xml), 469-471 (live_hosts_from_xml); consumed in execute_auto:719-735

## Evidence
```
open_ports_from_xml: `try:\n    root = ET.parse(path).getroot()\nexcept ET.ParseError:\n    return []` and identically in live_hosts_from_xml. execute_auto then does `tcp_ports = open_ports_from_xml(discovery_xml,'tcp')` (720) and `live_hosts = live_hosts_from_xml(discovery_xml) or None` (722); empty -> append_skipped_stage(tcp_identify, 'no open TCP ports') (729) and append_skipped_stage(udp_identify, 'no live hosts') (753). finish_plan sets status='done'. I reproduced it end-to-end: a discovery XML truncated mid-port (nmap killed mid-write) that still contained `<status state="up"/>` and port 22 produced: exit 0, status='done', both identify stages skipped, manifest import_xml_files=[].
```
## Impact
If nmap is OOM-killed, disk fills, the process is interrupted after partial XML flush, or any malformed XML is produced, the scanner silently reports a successful scan that found nothing. An internal host actually exposing SSH/RDP/etc. is recorded as having zero open ports and is dropped from the import set. For a production exposure scanner this is the worst failure mode: a broken scan is indistinguishable from a clean network, so real exposure goes unnoticed.

## Proposed fix
Distinguish parse failure from genuinely-empty. Have open_ports_from_xml/live_hosts_from_xml raise (or return a sentinel) on ET.ParseError instead of []. In execute_auto, treat a discovery whose XML is missing/unparseable as a stage failure (record returncode!=0 / status!='done') rather than a clean skip. Also sanity-check that the discovery .xml exists and parses before deciding 'no ports/no hosts'.

## Verifier rationale
Confirmed in code. open_ports_from_xml (scanner/scanops_scanner.py:445-448) and live_hosts_from_xml (:468-471) both `return []` on ET.ParseError, making a corrupt/unparseable discovery XML indistinguishable from a genuinely empty result. execute_auto consumes these at :720 (tcp_ports) and :722 (live_hosts); empty tcp_ports -> append_skipped_stage(tcp_identify) at :729; empty live_hosts -> skip udp_identify at :752-753. append_skipped_stage records returncode:0, skipped:True (:521-522). finish_plan sets status='done' (:655); write_manifest builds import_xml_files only from non-skipped rc==0 runs (:631-637) so it is empty; execute_auto returns 0 (:769). The GUI keys solely off this exit code and shows '완료' (gui:402), so a broken scan is presented as a clean, successful one. No post-scan validation flags 'discovery saw a live host but identify produced nothing'. This is distinct from ISSUE-001 (nonzero UDP rc aborts whole plan); here it is the inverse — a failure that should abort is silently swallowed.

One correction to the reporter's framing: the OOM-kill / SIGKILL / disk-full subset is largely MITIGATED, because run_nmap_stage records subprocess.call's rc (:674) and a killed nmap usually returns nonzero, hitting fail_plan at :716-717 — those cases fail loudly, not silently. So the impact's 'OOM-killed/interrupted' examples are overstated. However the CORE defect is genuine and reachable: nmap can exit 0 while emitting XML that ElementTree rejects (illegal XML/control bytes in service-fingerprint or NSE script-output fields is documented nmap behavior).
