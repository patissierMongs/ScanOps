# QA-025 — GUI 'TCP만'/'단일 실행에 UDP 추가'/NSE toggles silently no-op or conflict depending on mode, with no feedback

- **group**: Z    **disposition**: DEFER    **severity**: low    **dimension**: ux-gaps
- **dedup_key**: `gui-mode-irrelevant-toggles-silent-noop-and-open-closed-conflict`
- **location**: scanner/scanops_scanner_gui.py:154-158 (checks), 165-166 (NSE), _command:313-324; CLI semantics scanner/scanops_scanner.py:298 (args.udp only matters when -sU absent) and build_base_flags vs apply_auto_modifiers

## Evidence
```
The '단일 실행에 UDP 추가' checkbox label itself admits it only applies to single mode, but the UI still lets the user check it in auto mode where --udp is consumed by build_base_flags (only used for single profiles, scanner line 298) and ignored by the auto path (execute_auto/build_auto_flags never read plan['udp']/args.udp). Likewise '단일 실행에서 기본 NSE 추가' (nse_default, gui line 165) is meaningless in auto mode. open_only and include_closed can both be checked simultaneously (gui lines 156-157); the CLI applies include_closed (strip --open) then open_only (re-add --open) so include_closed silently wins/loses depending on order (scanner lines 314-317 / 376-379) with no warning.
```
## Impact
Users toggling these options in the wrong mode get no effect and no message, leading them to believe they constrained/expanded the scan when they did not (e.g. checking 'TCP만' is honored, but checking '단일 실행에 UDP 추가' in auto mode does nothing). Conflicting open/closed selections produce non-obvious behavior. This erodes trust and can cause an operator to ship a scan that doesn't match their intent.

## Proposed fix
Disable/grey out mode-irrelevant checkboxes based on the selected run mode (auto vs single), and make open_only/include_closed mutually exclusive radio-style. Add a brief warning when a checked option will be ignored for the current mode.

## Verifier rationale
Verified against code. The core claim holds: the GUI never gates its checkboxes by run mode (`_mode_changed` gui:197-214 only updates label/port-preset text; the Checkbuttons at gui:154-158,165-166 have no state management), and several toggles are genuine silent no-ops in auto mode.

CONFIRMED no-ops in auto mode:
- `--udp` (self.udp): `_command` always appends `--udp` if checked (gui:315-316) regardless of mode. In the CLI `args.udp` is read ONLY in build_base_flags (scanner:298), which only feeds plan["base_flags"]. The auto path builds stage flags via build_auto_flags (used whenever stage_id is set, scanner:417) and applies apply_auto_modifiers (scanner:368-380), which never reads any udp key. So checking "단일 실행에 UDP 추가" in auto mode does nothing.
- `--nse-default` (self.nse_default, gui:165/317-318): args.nse_default is read only in build_base_flags (scanner:310); the auto path uses plan["scripts"] (apply_auto_modifiers scanner:374) and AUTO_*_FLAGS carry their own NSE, so nse_default is ignored in auto.
- open_only and include_closed can both be checked at once (gui:156-157, no mutual exclusion) and both flags are emitted (gui:321-324) with no warning.

There is no feedback anywhere when a checked option is irrelevant for the current mode. This can cause an operator to believe they expanded coverage (e.g. added UDP in auto) when they did not — a real UX/intent-correctness defect for a tool aimed at production internal-exposure scanning, distinct from ISSUE-001 (exit-code abort).
