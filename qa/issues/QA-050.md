# QA-050 — open_host_ports_from_xml records ("", port) for a MAC-only host, so open_tcp/open_udp can be >0 while live_hosts=0 (summary internal disagreement)

- **group**: R4   **disposition**: FIX   **severity**: low   **dimension**: result-reliability
- **dedup_key**: `open-host-ports-empty-ip-mac-only-host`
- **location**: scanner/scanops_scanner.py open_host_ports_from_xml (no `if not ip` guard) vs hosts_with_open_ports_from_xml / live_hosts_from_xml (both guard `if ip and ...`)

## Evidence
`open_host_ports_from_xml` leaves `ip=""` when a `<host>` has only a MAC `<address>` and still records `("", pid)` — it never requires a non-empty ip. The two sibling helpers guard with `if ip and ...`. On a MAC-only host XML with tcp/22 open: `open_host_ports_from_xml → [("",22)]` (counted in open_tcp) but `hosts_with_open_ports_from_xml → []` and `live_hosts_from_xml → []` (live=0). `scan_findings` then reports open_tcp≥1 with live_hosts=0 — a self-contradictory summary.

## Proposed fix
Add `if not ip: continue` after the address loop (before the port loop), mirroring the existing guards, so the three counts stay consistent. Add a MAC-only-host test (QA-039's test uses only ipv4 hosts).

## Notes
A QA-039 helper gap (introduced in round 3). Distinct from QA-030 (inverse: live undercount). Low severity (MAC-only host with scanned ports is uncommon) but trivially triggerable and cheap to close. Verified in code.
