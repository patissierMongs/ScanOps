# QA-053 — expand_targets enforces --max-hosts on the pre-dedup running list, spuriously rejecting valid input that dedups within the cap

- **group**: R4   **disposition**: FIX   **severity**: low   **dimension**: scan-procedure / robustness
- **dedup_key**: `expand-targets-cap-before-dedup`
- **location**: scanner/scanops_scanner.py expand_targets — in-loop `if len(hosts) > cap` checks the accumulated (pre-dedup) list; dedup (`dict.fromkeys`) only happens after the loop

## Evidence
`expand_targets` accumulates into a list and caps the running length, but dedups only at the end. Duplicate/overlapping targets inflate the running count and trip the cap even though the deduped result fits. Reproduced: `expand_targets(['10.0.0.1','10.0.0.1','10.0.0.2','10.0.0.3'], cap=3)` raises despite 3 unique hosts; `expand_targets(['10.0.0.0/30','10.0.0.0/30'], cap=4)` raises despite 4 unique hosts. Runs in both batched and non-batched modes (create_plan always calls expand_targets for the cap check). The error ("too many hosts, adjust --max-hosts") is misleading when the unique count is within cap — a footgun for the exact QA-018 overlapping-scope scenario.

## Proposed fix
Accumulate into a dict (dedup as we go) and cap on the deduped running count, preserving the per-CIDR `num_addresses > cap` pre-check (QA-015 — must NOT be removed; it bounds a single oversized CIDR before materialization). This fixes QA-053 while keeping cross-target accumulation bounded.

## Notes
Not covered by QA-015 (single oversized CIDR) or QA-018 (added dedup). The existing dedup test uses cap=100, far above the count, so it never exercises the in-loop cap with duplicates. Verified by execution.
