# QA-047 — Force-killed stop shows "실패(종료 코드 -9) — 재개할 상태가 없습니다", labeling a deliberate stop as failure and falsely denying a resumable state

- **group**: R4   **disposition**: FIX   **severity**: medium   **dimension**: ui-ux
- **dedup_key**: `final-status-negative-rc-shown-as-unresumable-failure`
- **location**: scanner/scanops_scanner_gui.py final_status_text (no `rc < 0` branch) + _drain_output (rc = int(payload), no normalization), reached via _force_kill SIGKILL/taskkill

## Evidence
When the graceful stop signal does not exit the CLI within 6s, `_force_kill` SIGKILLs the child; `proc.wait()` returns a negative value (-9). `final_status_text` only branches on rc∈{0,130}, has_resume, rc==2, so -9 falls through to `실패(종료 코드 -9) — 재개할 상태가 없습니다`. Wrong twice: (1) a user STOP is labeled failure; (2) the CLI wrote `state.json` (status='running') before the loop, so a resumable state + partial XML exist on disk — but the SIGKILLed CLI never printed a resume hint, so `_resume_hint` is empty and autofill is skipped.

## Proposed fix
Add a `rc < 0` branch in `final_status_text` returning a '중지됨' message that points the user at the `state.json 선택` button (state exists on disk). Add a negative-rc case to the existing test. (The cheap, clearly-correct half is the message; deriving/autofilling the state path from output_dir+name on force-kill is a larger optional follow-on, deferred to avoid duplicating safe_name.)
