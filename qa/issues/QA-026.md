# QA-026 — GUI never validates output name / batch-size numeric edge cases or surfaces CLI validation failures distinctly — silent acceptance of empty results from typos

- **group**: Z    **disposition**: DEFER    **severity**: low    **dimension**: ux-gaps
- **dedup_key**: `gui-no-messagebox-rc2-validation-vs-runtime-failure-indistinct`
- **location**: scanner/scanops_scanner_gui.py:325-329 (batch check) and _command 277-337; CLI argparse type=int at scanner/scanops_scanner.py:825-826

## Evidence
```
The GUI validates batch size only as `batch.isdigit()` (gui line 327), which rejects negatives by treating them as non-digit but gives the generic message "배치 크기는 숫자여야 합니다"; a value like `0` is allowed and silently means 'one nmap run' but the UI label '배치 크기' offers no hint of that meaning. More importantly, all other invalid inputs (bad ports, bad target tokens, max-hosts cap exceeded) are only caught by the CLI subprocess, whose `error: ...` text (scanner lines 838-843, rc=2) is dumped into the scrolling log as plain text with `exit code: 2`; there is no messagebox or status distinction, so a validation rejection looks the same as a generic failure.
```
## Impact
A user who fat-fingers a CIDR that expands beyond --max-hosts (default 65536) or an invalid port spec gets a Korean `error:` line buried in the log and a red-herring `종료 코드 2`, with no modal drawing attention. Combined with finding #2 (empty success), misconfigured runs are easy to misread as either success or an opaque crash.

## Proposed fix
Detect rc==2 (CLI ValueError convention) in _drain_output and pop a messagebox with the captured error line, distinct from other failures. Add a tooltip/label clarifying batch size 0 = single run, and pre-validate ports/targets client-side before spawning.

## Verifier rationale
Partially confirmed. The accurate core: scanner/scanops_scanner_gui.py:399-403 (_drain_output) collapses every exit code into a single generic "종료 코드 {rc}" status and just _append_log's output as plain text, with NO messagebox for any failure. There is no distinction between rc=2 (CLI ValueError from invalid ports/targets/max-hosts cap) and rc=1 (OSError/runtime). Grounded: CLI main() lines 838-840 returns rc=2 on ValueError; ValueErrors are raised for bad target tokens (line 177), invalid ports (line 185), and max-hosts cap exceeded (line 226). So a config-validation rejection genuinely looks identical to a generic crash, surfaced only as a buried Korean "error:" log line. Also confirmed: batch size is validated only via isdigit() (gui line 327), and "0"/empty silently means a single nmap run (CLI default, line 825) with no UI hint. HOWEVER, the finding's headline claim is misframed and NOT real: the GUI not validating the output name does NOT cause "silent empty results from typos" — the CLI's safe_name (lines 128-130) sanitizes the name (regex substitution + fallback to scan_<timestamp>), so a bad name is silently cleaned, never empty or failing. Severity is low: the error text is always printed to the visible log, so nothing is truly hidden — the defect is only that attention isn't drawn and a validation rejection is indistinguishable from a crash.
