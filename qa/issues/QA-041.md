# QA-041 — Resume trusts recorded run["files"] without an existence check: a succeeded stage whose XML was deleted is never re-run, and the manifest advertises nonexistent importable XML as "done"

- **group**: R3   **disposition**: FIX   **severity**: high   **dimension**: stability-hang
- **dedup_key**: `manifest-trusts-rc0-files-without-existence-check`
- **location**: scanner/scanops_scanner.py manifest_xml_files (rc==0 branch returns files directly, no exists() check), stage_succeeded (returns True from recorded returncode==0 only)

## Evidence
`manifest_xml_files` applies the existence/parse filter (`xml_has_hosts`) ONLY in the rc!=0 branch; for rc==0 it returns the recorded files verbatim. `run["files"]` is captured once at scan time and never re-validated, and `stage_succeeded` returns True purely from `returncode==0`, so on --resume a succeeded stage is not re-run. Reproduced: after a 'done' auto run, deleting the identify XMLs and `--resume` re-ran 0 stages, kept status=done, and listed both deleted XML paths in `import_xml_files`, while the summary printed `import_xml=2` — and `create_zip` (which gates on exists()) omitted them, so the manifest advertises XML that is neither on disk nor in the zip.

## Proposed fix
Filter the rc==0 branch of `manifest_xml_files` by `Path(p).exists()`. Have `stage_succeeded` treat a succeeded stage whose recorded files all vanished as not-succeeded, so resume re-runs it instead of advertising vanished XML.
