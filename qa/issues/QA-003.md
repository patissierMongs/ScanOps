# QA-003 — One bad host/stage in a single batch aborts ALL remaining batches in a multi-batch run

- **group**: A    **disposition**: FIX    **severity**: high    **dimension**: failure-isolation
- **dedup_key**: `batch-loop-aborts-remaining-batches-on-single-failure`
- **location**: scanner/scanops_scanner.py:711-766 (execute_auto batch loop) and :700-708 (execute_single)

## Evidence
```
for idx in range(int(plan["cursor"]), len(plan["batches"])):  # :712
    ... rc = run_nmap_stage(...)
    if rc != 0: return fail_plan(plan, state_path, rc)  # :716/717 etc — returns out of the batch loop entirely
```
## Impact
With --batch-size > 0 (the documented mode for scanning many hosts / large CIDRs), batches are independent units of work. A non-zero exit in batch 0 (e.g. one unreachable subnet, or one host running a service that crashes nmap) returns out of the loop, so every later batch is never scanned. The plan is 'failed' even though batches that already finished have good XML, and the bulk of the target range is silently never touched. For internal network exposure scanning this means large blind spots from a single flaky host.

## Proposed fix
Isolate failures at the batch level: wrap each batch's stages so a failure records the batch as failed and `continue`s to the next batch instead of returning. Advance plan['cursor'] past completed batches and finish with an aggregate status, so resume re-runs only the failed batches.

## Verifier rationale
Confirmed genuine and grounded in code. make_batches (scanner/scanops_scanner.py:230-233) produces multiple batches when --batch-size>0, which is documented (line 825) as the mode for scanning many hosts/large CIDRs. In execute_auto the batch loop `for idx in range(int(plan["cursor"]), len(plan["batches"]))` (line 712) executes `return fail_plan(plan, state_path, rc)` on every non-zero stage rc — at lines 717, 727, 749, 758, 763. Because it is a `return`, the first failing stage in ANY batch exits the whole loop, so all later batches (idx+1..N-1) are never run, and the plan is marked status="failed" (fail_plan, lines 692-697). execute_single has the identical defect (lines 701-708, return at 704). This is a distinct axis from ISSUE-001 (which is about a UDP stage failure discarding TCP results within a single batch); this is about cross-batch failure isolation in a multi-batch run.

Verified impact and one nuance the finding slightly overstates: because fail_plan returns before `plan["cursor"]=idx+1` (line 765) runs, cursor is NOT advanced past the failed batch, so a manual --resume (load_plan 586-598, execute restarts loop at plan["cursor"]) re-runs from the failed batch and, via stage_succeeded, skips already-completed stages. So it is not strictly "silently never touched forever" — recovery is possible.
