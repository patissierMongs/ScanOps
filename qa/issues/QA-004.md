# QA-004 — tcp_discovery rc!=0 aborts before any identify even when discovery wrote a usable host/port XML

- **group**: A    **disposition**: FIX    **severity**: medium    **dimension**: failure-isolation
- **dedup_key**: `tcp_discovery-nonzero-rc-aborts-plan-discards-usable-xml`
- **location**: scanner/scanops_scanner.py:714-717 (execute_auto)

## Evidence
```
if not stage_succeeded(plan, idx, "tcp_discovery"):
    rc = run_nmap_stage(plan, idx, state_path, "tcp_discovery")
    if rc != 0:
        return fail_plan(plan, state_path, rc)
```
## Impact
Discovery uses -sS raw-socket flags (AUTO_TCP_DISCOVERY_FLAGS, :54-59) plus aggressive timing (-T4, --max-parallelism 100). nmap can exit non-zero (e.g. RST ratelimit warnings escalating, a transient send failure) after having already written discovery XML with live hosts and open ports. Because tcp_discovery rc!=0 hard-returns, tcp_identify and udp_identify never run despite a perfectly usable discovery XML existing on disk (open_ports_from_xml/live_hosts_from_xml would parse it). The whole auto pipeline is lost over a discovery exit code that did not actually invalidate the discovery data.

## Proposed fix
After discovery, decide continuation on the parsed XML (live_hosts / tcp_ports present) rather than on rc alone. If discovery produced parseable host data, proceed to identify stages and record discovery as 'completed with warnings' instead of failing the plan.
