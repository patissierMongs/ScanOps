# QA-024 — Auto discovery stage keeps raw-socket-only probes (-PE/-PA/--defeat-rst-ratelimit) even under --scan-type connect

- **group**: Z    **disposition**: DEFER    **severity**: low    **dimension**: nmap-coverage
- **dedup_key**: `connect-scantype-keeps-rawsocket-discovery-probes`
- **location**: scanner/scanops_scanner.py:49-59 (AUTO_TCP_DISCOVERY_FLAGS: -PE, DISCOVERY_PA='-PA80,443,3389', --defeat-rst-ratelimit), applied via build_auto_flags:384-388 then apply_auto_modifiers/set_scan_type

## Evidence
```
Dry-run with --scan-type connect: `-sT -PE -PS21,22,... -PA80,443,3389 -n -T4 --reason ... --defeat-rst-ratelimit ...`. set_scan_type swaps -sS->-sT but leaves -PE (ICMP echo), -PA (raw TCP ACK ping) and --defeat-rst-ratelimit untouched. These probe types require raw sockets; unprivileged nmap cannot send raw ICMP/ACK and downgrades or warns, and --defeat-rst-ratelimit has no effect without raw sockets.
```
## Impact
The 'no privileges' path is not actually privilege-clean: discovery loses its ICMP-echo and ACK reachability probes when run unprivileged, so hosts that answer only ICMP or only ACK (the exact cases the -PE/-PA combo at :47-48 was added to catch) are missed, and discovery silently behaves differently than its documented design. Coverage degrades without any signal to the operator.

## Proposed fix
Tie raw-socket probe flags to privilege availability: when scan_type=='connect', build a connect-compatible discovery probe set (e.g. -PS only, drop -PE/-PA/--defeat-rst-ratelimit) and note in preview/log that ICMP/ACK discovery is unavailable, or warn that auto discovery assumes privileged scanning.
