# QA-048 — Single-workflow --tcp-only / --scan-type connect strip -sU but leave U: ports in a user --ports override, producing a command nmap fatally rejects

- **group**: R4   **disposition**: FIX   **severity**: medium   **dimension**: scan-procedure
- **dedup_key**: `tcp-only-connect-leave-udp-ports-in-override`
- **location**: scanner/scanops_scanner.py build_base_flags — tcp_only U:-strip only touches the PRESET -p; the user --ports override is applied afterward and never filtered; connect strips only -sU

## Evidence
The --tcp-only branch rewrites U: out of the preset -p via `tcp_only_ports`, but the user override -p is set later and never passes through it. `--workflow single --profile basic --tcp-only --ports T:80,U:53` → `-p T:80,U:53` with no -sU → nmap fatally aborts ("UDP port specified but no UDP scan was requested ... QUITTING"). Same via connect: `--scan-type connect --ports U:53` → `-sT ... -p U:53`. Contradicts the --tcp-only help ("Remove UDP scan and U: ports"). The auto-workflow guard does not cover the single workflow. The GUI passes these straight through.

## Proposed fix
After the override -p is set, if `tcp_only` or `scan_type=='connect'`, run `tcp_only_ports` on the final -p value; if the stripped result is empty, raise a clear ValueError (mirroring the auto guard) so `--tcp-only --ports U:53` is rejected up front instead of failing inside nmap.

Not covered by QA-037 (presets only; explicitly scoped out user --ports), QA-010 (connect -sU; its test uses no --ports override), QA-036 (opposite symptom: silently dropping UDP in auto). Verified by execution.
