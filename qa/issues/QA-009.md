# QA-009 — GUI stop (taskkill /F /T) force-kills the scanner, so state.json is left permanently as status=running (zombie state)

- **group**: D    **disposition**: FIX    **severity**: high    **dimension**: stability-hang-stop
- **dedup_key**: `gui-stop-forcekill-leaves-state-running-zombie`
- **location**: scanner/scanops_scanner_gui.py:380-390 (_stop); scanner/scanops_scanner.py:780-792 (execute: status=running set, only KeyboardInterrupt writes status=interrupted)

## Evidence
```
GUI _stop: subprocess.run(["taskkill", "/F", "/T", "/PID", str(self.proc.pid)] ...) (line 386). The scanner sets plan["status"]="running"; write_json(state_path, plan) at scanner lines 780-781, and ONLY rewrites it to "interrupted" inside `except KeyboardInterrupt` (lines 787-790). taskkill /F is SIGKILL-equivalent: no Python exception is raised, so the except block never runs. Reproduced empirically: after taskkill /F /T the scanner exited with code 1 (not 130) and the state file's status stayed "running" indefinitely.
```
## Impact
Every time a user clicks the GUI 중지 (Stop) button, the saved state is left as "running" forever. There is no later process updating it. A human (or any downstream tooling) reading the state file cannot distinguish a still-running scan from a stopped one. The interrupted/resume UX the code advertises ("Resume with: --resume ...", scanner line 791) never prints and the status is wrong.

## Proposed fix
Make GUI stop graceful: on Windows send CTRL_BREAK_EVENT (self.proc.send_signal(signal.CTRL_BREAK_EVENT)) to the CREATE_NEW_PROCESS_GROUP child so the scanner's KeyboardInterrupt path runs and writes status=interrupted, with a short wait()+taskkill /F /T fallback only if it doesn't exit. On POSIX use SIGINT (not SIGTERM) for the same reason. Alternatively, have the scanner install a SIGTERM/SIGBREAK handler that writes status=interrupted before exiting.

## Verifier rationale
Verified against the actual code. GUI _stop force-kills via taskkill /F /T (scanner/scanops_scanner_gui.py:386) on Windows and os.killpg(pid, SIGTERM) on POSIX (line 388). taskkill /F is TerminateProcess (SIGKILL-equivalent) and raises no exception in the target; POSIX SIGTERM is also uncaught. The scanner sets plan["status"]="running" (scanner/scanops_scanner.py:780-781) and ONLY rewrites it to "interrupted" inside `except KeyboardInterrupt` (lines 787-790, return 130). I confirmed the scanner has NO `import signal` (imports at lines 8-22) and NO signal.signal handler anywhere, and main() (lines 833-843) installs none. Therefore the force-kill path never reaches any cleanup: status stays "running" permanently. finish_plan writes "done" (655) and fail_plan writes "failed" (693) only on normal completion/error — neither runs. No later writer corrects it: the GUI _drain_output handler (lines 392-406) only updates the in-GUI status label, never state.json. Net effect: every click of the GUI Stop button leaves a zombie state.json with status="running", indistinguishable from a live scan, and the advertised "Resume with: --resume" hint (line 791) never prints. This is distinct from ISSUE-001 (UDP exit aborting the plan). Two minor corrections to the finding: (1) the bug is NOT Windows-only — the POSIX SIGTERM path is equally uncaught and also leaves "running"; (2) the exact observed exit code (claimed 1) is not code-verifiable but is immaterial to the defect. The proposed fix is viable because the child is already launched with CREATE_NEW_PROCESS_GROUP (line 367) / start_new_ses
