# QA-054 — QA-047 fix is POSIX-only: Windows force-kill (taskkill /F) returns rc=1, so a deliberate stop still shows "실패 — 재개할 상태가 없습니다" and denies resume

- **group**: R5   **disposition**: FIX   **severity**: medium   **dimension**: ui-ux
- **dedup_key**: `qa047-posix-only-windows-forcekill-rc1-mislabeled`
- **location**: scanner/scanops_scanner_gui.py final_status_text (rc<0 branch is POSIX-only) + _force_kill (taskkill on Windows) + _drain_output

## Evidence
The round-4 QA-047 fix added an `rc < 0` branch so a force-killed stop reads as '중지됨'. But a negative rc is a POSIX-only subprocess convention. On Windows `_force_kill` uses `taskkill /F /T`, after which `proc.wait()` returns **1** (positive). So on Windows the `rc < 0` branch is unreachable: a force-killed scan yields rc=1 with no resume hint, so `final_status_text(1, ..., has_resume=False)` falls through to `실패(종료 코드 1) — 재개할 상태가 없습니다` — the exact QA-047 symptom re-appearing on the GUI's primary platform. The CLI did write `state.json` (status='running') before the loop, so a resumable state exists. The QA-047 test only feeds -9, never rc=1.

## Proposed fix
Track a `self._user_stopped` flag (set in `_stop`, reset in `_start`) and pass it to `final_status_text`; when set (or rc<0 or rc==130), render a '중지됨 — state.json 선택 후 재개' message regardless of the positive rc. Add a (rc=1, user_stopped) test case. A genuine rc=1 failure (not user-stopped) still shows '실패'.
