# QA-023 — No guard against resuming/running a state that is still status=running — two scanners can write the same state file and outputs

- **group**: I    **disposition**: FIX    **severity**: low    **dimension**: stability-hang-stop
- **dedup_key**: `no-status-guard-on-resume-or-rerun-of-running-state`
- **location**: scanner/scanops_scanner.py:586-598 (load_plan) and 772-786 (execute); no status check anywhere before re-running

## Evidence
```
load_plan() validates only plan['tool']=='scanops_scanner' (line 589) and nmap path; it never inspects plan['status']. execute() unconditionally sets status='running' and proceeds (lines 780-786). Because a hard-stopped scan is left as status='running' (see other finding), a user who clicks 재개 실행 (Resume) — or starts a fresh run pointing at the same name/output-dir — gets a second scanner that re-uses the same state_path and -oA output bases (output_base, lines 404-411) with no lock or staleness check.
```
## Impact
If the original scanner is somehow still alive, or if the user resumes a genuinely-running plan, two processes interleave writes to <name>.state.json (write_json is a plain truncate-write, line 530-531) and overwrite each other's nmap output files, producing corrupted state and lost/garbled results. Even in the common single-process case, the user gets no warning that the state was left 'running' rather than cleanly finished.

## Proposed fix
On resume, warn (or require --force) if status=='running'; on a fresh run, refuse to clobber an existing state_path whose status is 'running' unless forced. Optionally write a PID/heartbeat into the state and a simple lock file to detect a live concurrent scanner.
