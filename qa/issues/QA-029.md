# QA-029 — Interrupt/resume regression test is non-hermetic: it depends on a real nmap on the host, so the suite's pass/fail varies by machine

- **group**: R2    **disposition**: FIX    **severity**: low    **dimension**: verification-reliability
- **dedup_key**: `non-hermetic-interrupt-test-requires-real-nmap`
- **location**: backend/tests/test_standalone_scanner.py:845 (`test_interrupt_writes_interrupted_state_and_resume_hint`), specifically the `scanner.create_plan(args)` call at :858 with `--nmap nmap` and no `_fake_nmap`

## Evidence
```
>           plan = scanner.create_plan(args)
...
>               raise ValueError("nmap 을 찾을 수 없습니다. PATH 에 추가하거나 --nmap 경로를 지정하세요.")
E               ValueError: nmap 을 찾을 수 없습니다...
scanner/scanops_scanner.py:672: ValueError
```
The test passes `--nmap nmap`; `find_nmap("nmap")` (scanner:154) only accepts it if `Path("nmap").is_file()` or `shutil.which("nmap")` resolves it. On a host with nmap in PATH the test passes; on a host without nmap (headless CI, this container) `create_plan` raises before the interrupt path is ever exercised. Every other end-to-end test injects `_fake_nmap`; this one does not, so it is the only host-dependent test in the file.

## Impact
The QA-009 interrupt-handling guarantee (status=interrupted + resume hint on stop signal) is verified by a test whose result depends on the machine, not the code. A green run on a dev box with nmap gives false confidence; the same commit "fails" on a clean CI runner. This directly undermines "검증까지 완벽하게" — the regression net for graceful-stop is not reproducible.

## Proposed fix
Make the test hermetic without changing what it asserts: monkeypatch `scanner.find_nmap` to return a stub path (e.g. `"nmap"`) so `create_plan` succeeds, since the test already monkeypatches `subprocess.call` to raise `KeyboardInterrupt` and never actually launches nmap. (Equivalent: point `--nmap` at the `_fake_nmap` fixture.) The interrupt cleanup path — the actual thing under test — is then exercised on any host.

## Verifier rationale
Confirmed by direct repro: isolating this test on a host without nmap fails at create_plan, not in the interrupt logic. The scanner's `find_nmap` behavior is correct (it should not accept a non-existent binary), so the defect is in the test's setup, and the fix belongs in the test. Low severity (test-only) but real for verification reproducibility. Minimal monkeypatch, no production change — not overengineering.
