# QA-005 — Partial-but-valid XML from a non-zero-exit stage is written to disk but silently excluded from the import manifest

- **group**: A    **disposition**: FIX    **severity**: medium    **dimension**: failure-isolation
- **dedup_key**: `manifest-excludes-nonzero-exit-xml`
- **location**: scanner/scanops_scanner.py:624-637 (write_manifest), produced files recorded at :685 (run_nmap_stage)

## Evidence
```
run_nmap_stage records files unconditionally: "files": existing_outputs(base) at :685 (regardless of rc).
write_manifest then filters them out: all_xml_files / import_xml_files both require `if run.get("returncode") == 0 and not run.get("skipped")` (:627, :634).
```
## Impact
nmap routinely exits non-zero on RECOVERABLE conditions (e.g. host becomes unreachable mid-scan, an NSE script raising, '0 hosts up') while still writing a complete, useful XML for the hosts it did scan. Because the manifest's import list keys purely off returncode==0, that XML — which physically exists in output_dir and even ends up in the zip via create_zip (:645 iterates all runs' files) — is invisible to the ScanOps import path. Operators get a zip whose manifest omits real findings, so exposed ports go unreported.

## Proposed fix
Decide inclusion based on whether a parseable XML with host data was produced, not solely on rc. e.g. include a run's .xml in import_xml_files when ET.parse succeeds and it contains at least one <host>, even if rc!=0; optionally tag it as partial. Reuse the existing open_ports_from_xml/live_hosts_from_xml parse guards.

## Verifier rationale
Confirmed against scanner/scanops_scanner.py. run_nmap_stage records "files": existing_outputs(base) unconditionally at :685 and appends the run (with its non-zero returncode from :682) to plan["runs"] at :687 before returning rc. write_manifest then gates BOTH all_xml_files (:627) and import_xml_files (:634) on run.get("returncode") == 0, so a stage that exits non-zero but wrote a parseable, host-bearing XML is excluded from the import manifest. This path is genuinely reachable: every rc!=0 triggers an immediate return fail_plan(...) (execute_auto :717/:727/:749/:758/:763, execute_single :704), and fail_plan calls write_manifest(plan) at :696 with the failed run present. nmap commonly exits non-zero while still emitting a complete XML for hosts it scanned (host unreachable mid-scan, NSE script error, 0 hosts up), so real open-port data in output_dir becomes invisible to the ScanOps import list. The parse guards to do this safely already exist (open_ports_from_xml :442 / live_hosts_from_xml :464 both wrap ET.parse and return [] on ParseError). 

Two caveats on the finding as written: (1) the impact's claim that the XML "even ends up in the zip via create_zip" is INCORRECT for the failure path — fail_plan does NOT call create_zip; only finish_plan zips (:660-661), and finish_plan is unreachable while a non-zero run exists. So on failure no zip is produced at all; the XML sits in output_dir unreferenced.
