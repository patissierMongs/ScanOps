# QA-040 — scan_findings never counts open TCP ports found by tcp_discovery, so open_tcp drops to 0 when identify is skipped/fails after a successful discovery

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: result-reliability
- **dedup_key**: `scan-findings-ignores-discovery-open-tcp`
- **location**: scanner/scanops_scanner.py scan_findings (tcp_discovery branch only updates `live`, never the tcp set)

## Evidence
In `scan_findings` the tcp_discovery branch does only `live.update(live_hosts_from_xml(path))`; `open_tcp` is populated solely from the identify/single branch. So open TCP ports that discovery actually found are invisible unless identify also runs AND succeeds AND re-observes them. In the discovery-succeeds/identify-fails repro the discovery XML had tcp/22 open yet the summary printed `open_tcp=0` — an operator reads "host up, no open TCP ports", contradicting evidence the scanner already collected (execute_auto itself derives identify targets from those same discovery open ports).

## Proposed fix
In the tcp_discovery branch of scan_findings, also fold discovered open ports into the tcp set as a floor: `tcp.update(open_host_ports_from_xml(path,'tcp'))`. Since the set dedups, it cannot inflate above the true open set and only prevents under-reporting. (Pairs with QA-039 / QA-038.)
