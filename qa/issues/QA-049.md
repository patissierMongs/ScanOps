# QA-049 — A rc=0 host-less identify XML suppresses the QA-038 discovery fallback, so the manifest advertises an empty XML while the real live host/open ports are dropped (summary/manifest disagreement)

- **group**: R4   **disposition**: FIX   **severity**: high   **dimension**: result-reliability
- **dedup_key**: `hostless-rc0-identify-xml-suppresses-discovery-fallback`
- **location**: scanner/scanops_scanner.py manifest_xml_files (rc==0 branch filters by exists() but not xml_has_hosts) + importable_xml (fallback fires only when `seen` is empty)

## Evidence
QA-041 changed the rc==0 branch of `manifest_xml_files` to existence-filter only (no `xml_has_hosts`, unlike the rc!=0 branch). `importable_xml` adds the discovery fallback only when `seen` is empty. So when `tcp_identify` exits 0 but writes a host-less XML (live hosts go down/block between stages under -Pn), that empty XML stays in `seen` and SUPPRESSES the fallback. Reproduced (FAKE_NMAP_EMPTY_STAGE=tcp_identify --tcp-only): status=done, `summary live_hosts=1 open_tcp=2 import_xml=1`, but `import_xml_files=['...tcp_identify.xml']` (has_hosts=False) — the discovery XML with the real live host + open ports is excluded. A downstream import gets nothing on a 'done' scan. The rc!=0 case correctly salvages discovery; only rc==0 is broken.

## Proposed fix
Filter the non-discovery collection in `importable_xml` through `xml_has_hosts` (both passes), so a host-less identify XML never occupies `seen` and the fallback fires whenever no identify XML actually contains hosts.
