# QA-037 — tcp_only_ports truncates the spec at the first 'U:', dropping every later T: port (latent; masked by current presets)

- **group**: R3   **disposition**: FIX   **severity**: low   **dimension**: scan-procedure
- **dedup_key**: `tcp-only-ports-truncates-at-first-U`
- **location**: scanner/scanops_scanner.py:377-387 (tcp_only_ports), the `u_idx = find('U:'); port_spec = port_spec[:u_idx]` slice; called from build_base_flags under --tcp-only

## Evidence
`tcp_only_ports` slices off everything after the first `U:`, including later `T:` segments. Verified: `tcp_only_ports('U:53,T:80')==''`, `tcp_only_ports('U:53,T:80,443')==''`, `tcp_only_ports('T:80,U:53,T:443')=='T:80'`. The per-item loop below already correctly skips `U:`-prefixed items, so the head-truncation is redundant and wrong. Currently latent: the only preset with `U:` is phase1 (`T:1-65535,U:...`, all T: before U:), so truncation coincidentally yields the right value, and user `--ports` bypasses tcp_only_ports.

## Proposed fix
Delete the `u_idx`/slice block; keep only the per-item loop, which preserves all T:/bare-TCP items in any order while dropping U: segments. Add a unit test for interleaved specs.

## Notes
Different function/root cause from QA-013 (protocol_ports sticky). A strict simplification removing provably-wrong dead logic; guards against a future preset that interleaves U: before T:. Verified by execution.
