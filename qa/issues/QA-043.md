# QA-043 — write_json failure mid-scan (disk full / read-only dir) propagates OSError out of execute() uncaught, leaving status='running' permanently; writes are non-atomic

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: stability-hang
- **dedup_key**: `nonatomic-write-json-strands-running-status`
- **location**: scanner/scanops_scanner.py execute (only `except KeyboardInterrupt`); write_json (plain truncate-write); called mid-loop in run_nmap_stage / execute_single / execute_auto

## Evidence
`execute()` writes status='running' then runs the loop with only a `KeyboardInterrupt` handler. `write_json` is a plain `path.write_text(...)` invoked many times mid-loop. If any raises OSError (ENOSPC/EROFS/quota), it unwinds past finalize_plan (never runs) and is caught only by main()'s `except OSError` as rc=1, leaving on-disk state.json at status='running' permanently; a failed mid-write can also corrupt state.json.

## Proposed fix
Make `write_json` atomic: write `<path>.tmp` then `os.replace`. Broaden execute()'s handler so a non-KeyboardInterrupt exception during the loop downgrades status to 'interrupted'/'failed' and emits a resume hint (best-effort) before returning a failure code. Atomic writes also harden every other write_json caller.

## Notes
Not covered by QA-009 (force-kill, no Python exception), QA-023 (concurrency guard for existing running state; mentions non-atomic write only in passing), QA-017 (KeyError messaging; OSError is already caught by main but not handled for status). Verified in code.
