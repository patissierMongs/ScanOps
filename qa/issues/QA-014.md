# QA-014 — validate_ports accepts malformed specs (lone 'T:'/'U:', empty/double commas) that flow straight into nmap -p as garbage

- **group**: H    **disposition**: FIX    **severity**: medium    **dimension**: result-correctness
- **dedup_key**: `validate_ports_charclass_only_lets_malformed_specs_reach_nmap_p`
- **location**: scanner/scanops_scanner.py:97 (PORTS_RE), 180-186 (validate_ports); used at build_base_flags:301-304 and create_plan:550

## Evidence
```
PORTS_RE = `^[0-9TUtu:,\-\s]+$` only checks the character class, not structure. Verified: validate_ports('U:')='U:', validate_ports('T:')='T:', validate_ports(',,')=',,', validate_ports('T:80,,U:53')='T:80,,U:53'. build_base_flags then emits e.g. `-p U:` (verified: ['-Pn','-sV','-T4','-p','U:']). Note also `-p U:` is paired with a profile that has no -sU, so nmap is given a UDP port spec with no UDP scan type.
```
## Impact
Typos like `U:`, `,,`, or `T:80,,U:53` pass validation and are handed to nmap, which will either error out (wasting a scan run / failing the whole plan) or behave unexpectedly. Validation gives false confidence that the port spec is well-formed when it only filters characters.

## Proposed fix
Tighten validation to reject empty segments and dangling protocol prefixes: split on ',', require each non-empty segment to be a port/range optionally prefixed by exactly one T:/U:, and error on empty segments. Reuse the same parse logic that protocol_ports relies on so validation and consumption agree.
