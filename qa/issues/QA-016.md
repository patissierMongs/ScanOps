# QA-016 — IPv6 targets are accepted and expanded but the scan never gets -6, so nmap aborts (and in auto workflow aborts the whole plan)

- **group**: I    **disposition**: FIX    **severity**: medium    **dimension**: portability-edge
- **dedup_key**: `ipv6-targets-accepted-but-no-dash6-flag-nmap-fails`
- **location**: scanner/scanops_scanner.py:96 (TARGET_RE), 211-217 (expand_targets CIDR branch), 464-487 (live_hosts_from_xml), AUTO_*_FLAGS at 54-76 (no -6 anywhere)

## Evidence
```
TARGET_RE = `^[A-Za-z0-9_.:/\-]+$` includes `:` so `2001:db8::1` and `2001:db8::/64` pass validate_targets. expand_targets handles IPv6 networks via ipaddress.ip_network (line 213). live_hosts_from_xml re-validates extracted addrs with TARGET_RE (line 484), which also passes IPv6. But grep for `-6` across the file finds zero scan-flag usages — none of AUTO_TCP_DISCOVERY_FLAGS / AUTO_TCP_IDENTIFY_FLAGS / AUTO_UDP_IDENTIFY_FLAGS / PRESETS includes `-6`. nmap requires `-6` to scan IPv6 and errors out ("... you must specify -6" / cannot mix address families) when given an IPv6 target without it.
```
## Impact
Any IPv6 internal target is silently accepted at the CLI/GUI layer, then nmap fails at runtime. In the default auto workflow, the tcp_discovery stage exits nonzero, which (per the existing fail_plan behavior, ISSUE-001) aborts the whole plan. Operators auditing IPv6-enabled internal networks get no results and a confusing failure rather than an upfront 'IPv6 not supported' message.

## Proposed fix
Decide on support: either reject IPv6 targets early in validate_targets with a clear message, or detect IPv6 targets and inject `-6` into the nmap flags (noting nmap cannot mix IPv4/IPv6 in one invocation, so mixed batches must be split by family).
