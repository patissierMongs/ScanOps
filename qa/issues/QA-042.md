# QA-042 — Interrupt during finalize_plan (e.g. create_zip) overwrites a completed done/partial status with 'interrupted', emitting a misleading resume hint for a finished scan

- **group**: R3   **disposition**: FIX   **severity**: medium   **dimension**: stability-hang
- **dedup_key**: `late-interrupt-clobbers-terminal-status`
- **location**: scanner/scanops_scanner.py execute (except KeyboardInterrupt unconditionally sets status='interrupted'); finalize_plan writes terminal status then write_manifest/create_zip/print

## Evidence
`finalize_plan` writes the terminal status (done/partial/failed) then performs interruptible work (write_manifest, create_zip, prints), all inside execute()'s try block. A stop signal (SIGTERM/SIGBREAK→KeyboardInterrupt via install_stop_handlers, or Ctrl-C) during that window propagates to `except KeyboardInterrupt`, which unconditionally sets status='interrupted' and returns 130. Reproduced by making create_zip raise KeyboardInterrupt after all stages succeeded: final state.json was 'interrupted', rc=130, the zip half-written, and the user told to resume an already-complete scan. (The group-D stop-handler fix opened this window.)

## Proposed fix
In the except block, do not clobber a status that is already done/partial/failed: if `plan.get("status")` is terminal, return its proper exit code (0 for done/partial, 1 for failed) without rewriting. (Secondary: write the zip atomically — temp + rename.)

## Notes
Opposite of QA-009 (force-kill leaving status='running'); QA-027/022 are exit-code/hint UX. Verified in code; tiny scoped guard.
