# QA-051 — XML-only deletion defeats the QA-041 resume re-run: the .xml is dropped from the manifest but never regenerated because surviving .nmap/.gnmap siblings satisfy the vanish check

- **group**: R4   **disposition**: FIX   **severity**: medium   **dimension**: stability-hang
- **dedup_key**: `xml-only-deletion-defeats-resume-rerun`
- **location**: scanner/scanops_scanner.py stage_succeeded + rewind_cursor_for_vanished_outputs (vanish check keys on ANY recorded file) vs manifest_xml_files (keys on .xml)

## Evidence
The QA-041 vanish detection is asymmetric: `manifest_xml_files` filters on the `.xml` specifically, so an xml-only deletion drops it from the manifest. But the re-run half (`stage_succeeded`, `rewind_cursor_for_vanished_outputs`) checks `any(Path(f).exists() for f in files)` over ALL recorded files (.xml + .nmap + .gnmap from `existing_outputs`). When only the .xml is removed (an importer consumes it, or `rm *.xml` leaves the .nmap/.gnmap), the siblings keep `any(...exists())` True, so the stage is treated as succeeded and the cursor never rewinds. Reproduced (real CLI, delete only tcp_identify.xml): `--resume` exits 0 'done', the .xml is NOT regenerated, and `import_xml_files` silently loses it — permanent loss of importable coverage presented as a clean 'done'. The QA-041 regression test deletes all three suffixes, so it misses this.

## Proposed fix
Key the resume-side vanish check on `.xml` files only (the artifact the manifest advertises), in both `stage_succeeded` and `rewind_cursor_for_vanished_outputs`, with an `if xmls` guard to preserve behavior for runs that recorded no .xml. Add a regression test deleting only the .xml.
